From reward to weight update — the GRPO mechanic

How does a batch of 1/0 rewards actually become a gradient step on the LoRA weights? Four stages — this is GRPO's whole trick.

1. Group-relative advantage. Each prompt produces G completions, the verifier scores each with reward ri ∈ {0, 1}. Normalize within the group:

Ai = (ri − μgroup) / σgroup

Example — rewards [1, 0, 1, 0]: μ = 0.5, σ ≈ 0.577, so advantages ≈ [+0.87, −0.87, +0.87, −0.87]. The group's own mean is the baseline — no value/critic network. When all four rewards are 0, σ = 0 and every advantage is 0 — which is exactly the zero-reward-group case the next section is about.

2. PPO-clipped policy-gradient loss. For every token t in completion i, take the importance ratio against the pre-step policy:

ρt = πθ(tokt) / πθold(tokt)

Li,tpolicy = −min( ρt · Ai,  clip(ρt, 1−ε, 1+ε) · Ai )

Tokens in passing completions get their probability pushed up; tokens in failing completions get pushed down — scaled by the advantage. The clip caps how far a single step can move.

3. KL anchor to the reference model. Add a penalty for drifting from the frozen base OLMo (β = 0.05 in train.py):

Ltotal = Lpolicy + β · KL(πθ ‖ πref)

This is what the KL divergence chart below tracks — low and stable means the policy is learning without running away from the base distribution.

4. AdamW step on LoRA adapters only. Backprop Ltotal, but only the LoRA adapters on q/k/v/o/gate/up/down_proj have requires_grad=True (≈0.3% of parameters). The 7B base weights stay frozen. So one optimization step is: 4 prompts × G completions → verifier rewards → group-relative advantages → clipped PG loss + KL → one AdamW update on the LoRA matrices. Repeat 200 times.

Why the absolute gain is modest: sparse rewards at low baseline

The baseline model solves only 2.7% of problems. For GRPO to learn, each group of G completions needs at least one passing sample — otherwise every sample gets reward 0, the within-group advantage is 0, and the gradient contribution is 0.

With G = 4 generations per prompt and baseline pass rate p = 0.027, the fraction of groups that are entirely zero-reward is (1 − p)G = (1 − 0.027)4 ≈ 89.5%.

In other words: ~90% of optimizer steps carry no learning signal at all. Only the remaining 10% contribute gradient. The run is doing real RL, but on a fraction of the compute it looks like it's doing.

Counter-moves: more generations per prompt (G = 8 → 80% zero-reward · G = 16 → 64%), filter training set to problems the base model solves ≥ sometimes (cheapest, biggest lift), or warm-start with SFT to raise the baseline first. Tulu 3 does all three.

def is_undulating(num):
    # Function to check if a number is undulating
    # An undulating number is one where the difference between consecutive digits is either 1 or 0
    # For simplicity, this function assumes the input is a single-digit number, as undulating typically applies to sequences of digits
    if len(str(num)) == 1:
        return abs(str(num) - str(num - 1)) in [0, 1]
    else:
        # Check each digit in the number
        for i in range(len(str(num))):
            if abs(int(str(num))[i] - int(str(num)[i + 1])) not in [0, 1]:
                return False
        return True

def is_undulating(num):
    # Function to check if a number is undulating
    # An undulating number is one where the differences between consecutive digits are either 1 or 0
    # For simplicity, we assume the input is a non-negative integer

    # Convert the number to a string to handle digit manipulation easily
    num_str = str(num)

    # Check the differences between consecutive digits
    for i in range(1, len(num_str)):
        if abs(int(num_str[i]) - int(num_str[i-1])) != 1:
            return False

    return True
